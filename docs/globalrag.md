# GlobalRAG: 面向全局检索增强生成的语料库级推理基准

> 论文: [Towards Global Retrieval Augmented Generation: A Benchmark for Corpus-Level Reasoning](https://arxiv.org/html/2510.26205v2/)
> 作者: Qi Luo, Xiaonan Li, Tingshuo Fan, Xinchi Chen, Xipeng Qiu
> 机构: 复旦大学, 上海创新研究院
> 日期: 2025-10

## 1. 研究动机

### 1.1 核心问题

当前 RAG 基准聚焦于**局部 RAG**——检索相关文本块回答需要局部理解的问题。但真实应用需要**全局 RAG**——聚合整个文档集合的信息获取语料库级洞察。

**示例:** "这个数据库中引用最多的论文是哪些？"——需要遍历所有文档并聚合统计。

### 1.2 现有方法的失败

现有方法在全局任务上仅达到 **1.51 F1** 分数，揭示了架构级的根本局限。

### 1.3 三个根本问题

| 问题 | 描述 |
|------|------|
| 固定粒度分块破坏文档完整性 | 标准 512-token 分块机械切割文档，将元数据与值分离 |
| 密集检索返回语义相关但事实无关的噪声 | 检索到的文档消耗上下文窗口但不包含所需答案 |
| LLM 数值计算能力有限 | 模型在比较、遗漏和一致性方面表现差 |

## 2. GlobalQA 数据集

### 2.1 规模与特点

- **13,000+ 问答对**
- **2,000+ 简历**覆盖 23 个专业领域
- **42.6% 的查询需要 20+ 文档**（vs 现有基准的 1-4 个）

### 2.2 四种任务类型

| 任务类型 | 占比 | 示例 |
|----------|------|------|
| 计数（Counting） | 16.7% | "有多少人在X公司工作过？" |
| 极值（Max-Min） | 33.9% | "谁的工作年限最长？" |
| 排序（Sorting） | 16.3% | "按学历排列这些候选人" |
| Top-k 提取 | 33.9% | "列出薪资最高的5位员工" |

### 2.3 数据构建流程（逆向策略）

1. 自动生成查询配置
2. 执行轨迹获取确定性答案
3. 从完成的轨迹生成自然语言问题
4. 验证和过滤质量

### 2.4 评估指标

- **F1 分数:** Token 级精确率/召回率
- **Document F1@k (D-F1@k):** 衡量检索覆盖度

## 3. GlobalRAG 方法

### 3.1 三阶段流水线：检索 → 过滤 → 聚合

**阶段一：文档级检索（Document-Level Retrieval）**
- 将完整文档作为原子单位（vs 512-token chunk）
- 为每个文档构建 chunk 级索引
- 通过密集检索返回 top-k 最相似文档

**阶段二：文档级过滤（Document-Level Filter）**
- LLM 驱动的过滤机制
- 判断每个文档是否包含回答查询的信息
- 在推理前移除噪声

**阶段三：任务级聚合工具（Task-Level Aggregation Tools）**

| 工具 | 功能 |
|------|------|
| 计数工具 | 系统遍历、去重、实体枚举 |
| 极值工具 | 精确数值比较 |
| 排序工具 | 跨异构格式的一致排名 |
| Top-k 提取 | 基于堆的算法 + 剪枝 |

## 4. 实验结果

### 4.1 主要性能（Qwen2.5-14B-Instruct）

| 方法 | F1 | D-F1@20 |
|------|-----|---------|
| **GlobalRAG** | **6.63** | **12.01** |
| StandardRAG（最强基线） | 1.51 | 8.09 |
| IRCoT | 0.09 | - |
| FLARE | 0.55 | - |
| HyperGraphRAG | 0.09 | - |

**相对提升: 339%**（5.12 F1 绝对值提升）

### 4.2 跨数据集评估

方法在多跳基准上保持 12-41 F1，但在 GlobalQA 上坍塌至 <2 F1，揭示了 **95-99% 的相对下降**。

### 4.3 消融实验

| 配置 | F1 下降 |
|------|---------|
| Chunk-based 检索（vs Document-level） | **94.5%** |
| 移除过滤器 | 7.3% |
| 移除聚合工具 | **79.2%** |

### 4.4 检索步数影响

- GlobalRAG F1 从 0 步的 ~0 上升到 5 步的 7.0
- 继续增至 20 步仍有提升
- IRCoT 在 10 步后平台化

### 4.5 检索数量分析

- GlobalRAG 在 30 个检索文档时达峰
- IRCoT 在 10 个后因噪声累积而下降

### 4.6 模型规模影响

| 组件 | 小模型 → 大模型 | 提升 |
|------|----------------|------|
| 检索器 | BGE → Qwen3-8B | +18.7% |
| 过滤器 | Qwen3-4B → Qwen3-14B | +7.2% |

## 5. 核心贡献

1. **GlobalQA 基准:** 首个系统评估全局 RAG 能力的数据集（13,000+ QA 对，需分析 2-50 个文档）
2. **性能差距识别:** 证明现有方法仅达 1.51 F1，揭示架构级局限
3. **GlobalRAG 框架:** 通过文档级检索 + 智能过滤 + 符号计算工具实现 6.63 F1

## 6. 全局查询 vs 多跳推理

| 维度 | 多跳推理 | 全局聚合 |
|------|----------|----------|
| 文档数量 | 2-4 个文档链 | 20+ 文档并行 |
| 推理模式 | 串行链式 | 并行聚合 |
| 核心挑战 | 路径发现 | 统计计算 |
| 需要能力 | 关系推理 | 数值运算 + 去重 |

## 7. 核心洞察

GlobalRAG 揭示了 RAG 研究中**被忽视的盲区**：

1. **全局聚合是本质不同的任务** ——不是"更多跳"的多跳推理，而是质的不同。需要并行处理大量文档并执行统计操作。

2. **纯神经方法不够** ——LLM 在精确计数、比较和排序上天然不可靠，需要与符号计算工具结合。

3. **文档完整性至关重要** ——94.5% 的性能下降说明 chunk-level 分块对全局任务是致命的。

4. **混合架构是方向** ——神经检索 + 程序化执行的组合优于纯神经或纯符号方法。

**实践启示:** 企业 RAG 系统若需支持"统计型查询"（如报表生成、趋势分析），必须考虑文档级检索和程序化聚合工具，纯向量检索+LLM 生成无法满足需求。
