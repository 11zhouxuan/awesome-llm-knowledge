## 1. 系统定位

Sirchmunk 是一个**开源、无嵌入（embedding-free）的智能搜索引擎**，将原始数据实时转化为自进化的知识，无需向量嵌入、静态索引或复杂 ETL 管道。

### 1.1 三大设计支柱

| 支柱 | 原则 | 实现 |
|------|------|------|
| 无嵌入 | 以数据最纯粹的形式处理 | 无索引检索，通过智能文件发现实现零信息损失 |
| 自进化 | 数据是流，非快照 | 知识簇随每次搜索进化；LLM 驱动的自主深化理解 |
| Token 效率 | 最大化智能，最小化成本 | 仅在必要时触发 LLM；蒙特卡洛采样提取精确证据 |

### 1.2 与传统 RAG 的对比

| 维度 | Sirchmunk | 传统 RAG |
|------|-----------|----------|
| 启动成本 | 零 | 高（向量DB + 图DB + 解析器 + ETL） |
| 数据时效性 | 即时 | 滞后（需批量重新索引） |
| 信息保真度 | 完整 | 有损（向量近似） |
| 可扩展性 | 低资源消耗 | 线性成本增长 |
| 准确性 | 基于证据 | 近似匹配 |
| 知识进化 | 自进化 | 静态快照 |
| Token 效率 | 蒙特卡洛采样 | 完整文档块 |
| 故障恢复 | ReAct Agent 自适应 | 无补救机制 |

### 2.1 四层架构

### 2.2 两种搜索模式

| 模式 | 策略 | 耗时 | LLM 调用 | 适用场景 |
|------|------|------|----------|----------|
| FAST | 贪心策略 + 二级关键词级联 + 早停 | 2-5 秒 | ~2 次 | 简单查询 |
| DEEP | 完整管道 + 蒙特卡洛采样 + 多轮 ReAct | 10-30 秒 | 多次 | 复杂查询 |

FAST 模式比 DEEP 快 **10 倍**。

## 3. 核心搜索管道（五阶段）

### Stage 0: 知识复用

查询已存储的知识簇，语义相似匹配。命中则亚秒级返回；未命中进入完整管道。

### Stage 1: 并行探测

四路并发探测同时启动：
1. LLM 关键词抽取（多粒度）
2. 目录结构扫描（元数据）
3. 知识缓存查找
4. 指定路径上下文加载

### Stage 2: 检索与排序

两种互补策略并行：

| 策略 | 方法 |
|------|------|
| 内容检索 | 多级关键词 + IDF 加权评分 |
| 结构排序 | LLM 基于元数据（文件名、路径、预览）排序候选 |

**评分哲学:**
- 稀有性优先：高 IDF 词权重大于常见词
- 递减收益：词频通过对数饱和封顶
- 上下文质量：过短或过长段落受罚
- 精确性奖励：精确匹配 > 部分匹配

### Stage 3: 知识簇构建

- 文件路径合并去重
- **蒙特卡洛证据采样**从候选文档中提取相关区域（核心创新）
- LLM 将证据合成为结构化知识簇

### Stage 4: 输出与自适应

- 找到证据 → LLM 生成结构化 Markdown 简报
- 证据不足 → **ReAct 搜索 Agent** 激活，迭代探索

### Stage 5: 持久化

知识簇保存（含嵌入）以供未来复用。

## 4. 无索引检索引擎

### 4.1 核心理念

拒绝传统预处理范式（解析→分块→嵌入→存储），代之以：
- **ripgrep-all**（Rust 实现，支持 100+ 文件格式）进行实时全文搜索
- 智能采样替代全文读取

### 4.2 智能文件发现

| 技术 | 描述 |
|------|------|
| 结构线索 | 文件名、目录层次、扩展名携带强先验信号 |
| 多级关键词探测 | 问题分解为不同粒度的关键词（粗→细） |
| LLM 引导的结构排序 | 在深度读取前，LLM 基于元数据评估候选 |

## 5. 蒙特卡洛证据采样（核心创新）

### 5.1 解决的问题

候选文档可能有数百页，仅几段相关。完整阅读 token 成本过高；固定分块策略要么遗漏要么淹没在噪声中。

### 5.2 三阶段启发式算法

**Phase 1 — 撒网（Cast Wide Net）:**

| 策略 | 类型 | 方法 |
|------|------|------|
| 模糊锚定 | 利用（Exploitation） | Token 级模糊匹配，定位与查询和关键词相似的区域 |
| 分层随机采样 | 探索（Exploration） | 均匀间隔分层采样，确保无区域完全被忽略 |

每个采样窗口由 LLM 评分（0-10 相关性），附结构化推理。这是循环中**唯一的 LLM 调用**，操作于小窗口而非整个文档。

**Phase 2 — 聚焦（Focus）:**

高分样本成为"种子"。新样本从以种子为中心的**高斯分布**中抽取，sigma 跨迭代衰减（**模拟退火**方法）：
- 早期轮次：广泛探索
- 后期轮次：以递增精度收敛到最高价值区域
- **早停机制:** 样本超过高置信阈值时终止

**Phase 3 — 合成（Synthesize）:**

Top-K 评分片段传入 LLM，合成为连贯的兴趣区域（ROI）摘要。输出包含置信标志，指示是否找到强证据。

### 5.3 关键优势

- 不读取完整文档，仅采样关键区域
- 利用+探索平衡（类似多臂老虎机）
- 模拟退火式收敛，逐步精确
- LLM 调用最小化（仅对小窗口评分）

## 6. ReAct 搜索 Agent

当标准管道证据不足时，自动激活自主推理循环。

### 6.1 思维→行动→观察范式

```
循环:
  1. Thought: LLM 推理当前知识、缺失缺口、最有前途的行动
  2. Action: LLM 选择并调用工具（关键词搜索/文件读取/知识查询/目录扫描）
  3. Observation: 工具结果作为新上下文反馈
  → 重复直到找到答案或预算耗尽
```

### 6.2 工具优先级

1. 关键词搜索（最便宜，覆盖最广）
2. 文件读取（有前途的候选）
3. 知识缓存查询（已有洞察）
4. 目录扫描（其他方法失败时）

### 6.3 预算约束

双重预算防止无界循环：
- **Token 预算:** LLM 消耗总 token 硬限制
- **循环次数:** 最大推理-行动-观察循环数

Agent 维护已尝试操作的记忆，避免重复。

## 7. 自进化知识系统

### 7.1 知识簇（Knowledge Cluster）

基本单元，携带：

| 属性 | 说明 |
|------|------|
| 确定性身份 | 内容哈希，相同内容收敛到同一簇 |
| 证据链接 | 回溯到源文件和段落 |
| 置信分数 | 反映证据强度和一致性 |
| 抽象层次 | 技术→原理→范式→基础→哲学 |
| 生命周期状态 | 新生→稳定→争议/废弃 |
| 热度分数 | 访问频率 |

### 7.2 知识进化生命周期

```
新证据 → 无匹配 → 蒙特卡洛抽取 → LLM 合成 → 新簇 (新生)
                                                    ↓
相似查询 → 嵌入余弦相似度命中 → 复用 ← 反复验证 → (稳定)
                                                    ↓
底层数据变化 → 证据不再支持 → (争议/废弃)
```

### 7.3 知识图谱

语义边连接知识簇：

| 边类型 | 描述 |
|--------|------|
| 弱语义边 | 轻量加权连接，表示主题相关性 |
| 丰富认知边 | 类型化关系（路径、障碍、类比、捷径、解决方案） |

## 8. 存储与持久化

### 8.1 设计原则："默认快速，重要时持久化"

| 层 | 技术 | 用途 |
|----|------|------|
| 内存 | DuckDB | 微秒级读写，所有运行时数据 |
| 持久化 | Apache Parquet | 压缩、分析性能、生态兼容 |
| 原子写入 | 临时文件→原子重命名 | 崩溃一致性 |
| 嵌入存储 | 384维向量 | 知识簇相似度搜索（Stage 0） |

**注意:** 虽然号称"无嵌入"，系统实际在知识管理层使用轻量嵌入（384维）用于知识复用。"无嵌入"指的是检索阶段不依赖向量索引。

## 11. 核心洞察

Sirchmunk 在 LLM+知识库方向提出了一个**激进但务实的替代方案**：

### 11.1 "无嵌入"的真正含义

不是完全不用向量——而是**检索阶段不依赖预计算向量索引**。这解决了 RAG 的核心痛点：
- 无需预处理管道（解析→分块→嵌入→存储）
- 数据更新即时生效，无需重新索引
- 零信息损失（搜索原始文件，非近似向量）

### 11.2 蒙特卡洛采样是关键创新

将强化学习中的"探索-利用"范式应用于文档证据抽取：
- **Exploitation:** 模糊锚定已知相关区域
- **Exploration:** 分层随机采样避免遗漏
- **模拟退火:** 逐步收敛，平衡精度和覆盖度
- 仅对小窗口调用 LLM，极大降低 token 成本

### 11.3 知识的"新陈代谢"

知识簇的生命周期管理（新生→稳定→废弃）模拟了人类知识的新陈代谢——知识不是永恒不变的，需要通过反复验证成熟，通过数据变化淘汰。这比静态知识图谱更接近现实。

### 11.4 与其他方案的互补关系

| 场景 | 推荐方案 |
|------|----------|
| 大规模文档库，低延迟要求 | 传统 RAG（预计算索引） |
| 频繁变化的文档，精确回答 | **Sirchmunk**（实时搜索） |
| 结构化知识，需要推理 | GraphRAG |
| 单文档深度理解 | PageIndex / BookRAG |

### 11.5 实践启示

1. **对于中小规模文档集（如企业内部知识库），Sirchmunk 模式可能完全替代传统 RAG**——无需向量数据库基础设施
2. **蒙特卡洛采样思想可移植**到任何需要从长文档中精确定位信息的场景
3. **ReAct Agent 作为后备**是优雅的容错设计——标准管道失败时自动升级到更深度的探索
4. **自进化知识系统**实现了"越用越好"——频繁查询的知识加速响应，过时知识自动淘汰

**局限性:**
- 实时搜索在大规模文档集上延迟可能较高（10-30 秒）
- 对文件系统的依赖限制了分布式部署
- LLM 调用虽然优化，但仍是成本主要来源
- 知识簇的质量依赖 LLM 合成能力
