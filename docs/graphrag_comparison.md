# GraphRAG vs HippoRAG vs PathRAG vs OG-RAG: 知识图谱RAG架构选型研究

> 来源: [GraphRAG vs HippoRAG vs PathRAG vs OG-RAG: Choosing the Right Architecture for Your Knowledge Graph](https://medium.com/graph-praxis/graphrag-vs-hipporag-vs-pathrag-vs-og-rag-choosing-the-right-architecture-for-your-knowledge-graph-a4745e8b125f)
> 作者: Alexander Shereshevsky
> 发表日期: 2026-01-30

## 1. 研究背景

传统的向量检索（Vector RAG）在处理多跳推理、全局综合和结构化知识查询时存在明显局限。知识图谱增强的RAG架构通过引入实体关系、社区结构或本体约束来提升检索质量。本文比较了四种主流的图谱RAG架构，帮助选择适合不同场景的方案。

## 2. 四种架构详解

### 2.1 Microsoft GraphRAG

**核心方法:** 使用 Leiden 算法进行层次化社区检测（Hierarchical Community Detection），将知识图谱划分为不同粒度的社区，并对每个社区生成摘要。

**工作流程:**
1. 从文本中提取实体和关系构建图谱
2. 使用 Leiden 算法进行多层次社区检测
3. 对每个社区生成摘要描述
4. 查询时根据问题匹配相关社区，利用社区摘要生成回答

**核心优势:**
- 在摘要任务上有 72-83% 的提升
- 全局查询场景下可实现 97% 的 token 缩减
- 擅长语料库级别的综合与主题推理

**核心劣势:**
- 索引构建成本极高（大规模数据集每次重建约 $33K）
- 不适合需要频繁更新的场景

**最佳适用场景:** 语料库级别的综合分析和主题推理（如"这个领域的主要趋势是什么？"）

---

### 2.2 HippoRAG

**核心方法:** 受神经生物学启发，使用个性化 PageRank（Personalized PageRank）进行联想式检索，模拟人类海马体的记忆检索机制。

**工作流程:**
1. 构建知识图谱作为"皮层索引"
2. 使用 LLM 进行实体链接（类似海马体的模式分离/完成）
3. 通过个性化 PageRank 在图谱上进行联想式扩散检索
4. 检索相关子图生成回答

**核心优势:**
- 多跳推理成本降低 10-30 倍
- 速度比迭代方法快 6-13 倍
- 单次检索即可完成多跳推理（无需多轮LLM调用）

**核心劣势:**
- 需要高质量的连通图谱
- 依赖准确的实体链接
- 图谱连通性不足时效果下降

**最佳适用场景:** 成本敏感的多跳推理任务（如"A和B之间有什么关联？"）

---

### 2.3 PathRAG

**核心方法:** 基于流（Flow-based）的剪枝算法，提取知识图谱中可靠的关系路径，只保留与查询最相关的推理链路。

**工作流程:**
1. 构建知识图谱
2. 根据查询识别源节点和目标节点
3. 使用流算法计算节点间的最优路径
4. 剪枝不相关的路径，只保留高可靠性的关系链
5. 基于精炼后的路径生成回答

**核心优势:**
- 与 GraphRAG 头对头对比中 60% 的情况下胜出
- 实现 44% 的 token 缩减
- 生成的上下文更精准、噪声更少

**核心劣势:**
- 引入额外的计算开销（路径计算）
- 在关系链不清晰的场景下效果受限
- 需要明确的实体间关系路径

**最佳适用场景:** 上下文受限的应用（如"从A到B的因果链是什么？"）

---

### 2.4 OG-RAG (Ontology-Grounded RAG)

**核心方法:** 基于本体约束的超图（Ontology-grounded Hypergraph），使用预定义的 schema 约束知识表示和检索。

**工作流程:**
1. 定义领域本体（Ontology）和 schema
2. 基于本体约束构建超图结构
3. 查询时利用 schema 约束进行结构化检索
4. 确保回答符合本体定义的语义规范

**核心优势:**
- 回答正确性提升 40%
- 事实召回准确率提升 55%
- 强一致性和可解释性

**核心劣势:**
- 需要预定义本体，前期投入大
- 灵活性受限（用精度换取灵活性）
- 领域变化时需要更新本体

**最佳适用场景:** 强监管、重 schema 的领域（如医疗、金融合规、法律）

## 3. 关键对比维度

| 维度 | GraphRAG | HippoRAG | PathRAG | OG-RAG |
|------|----------|-----------|---------|--------|
| 查询类型 | 全局综合 | 多跳推理 | 路径推理 | Schema约束查询 |
| 成本 | 极高（索引） | 低 | 中等 | 中等（本体维护） |
| 精度提升 | 72-83%（摘要） | 多跳推理优势 | 60%胜率 vs GraphRAG | 40%正确性提升 |
| Token效率 | 97%缩减（全局） | 高效 | 44%缩减 | 高效 |
| 适用场景 | 主题分析 | 关联发现 | 因果推理 | 合规/医疗/金融 |

## 4. 查询复杂度与架构选择

- **单跳查询:** 图谱方法收益较小，向量检索即可满足
- **多跳推理:** 图谱方法可带来 2-3 倍准确率提升，HippoRAG 性价比最高
- **全局综合:** 需要社区摘要能力，GraphRAG 是首选
- **Schema约束查询:** OG-RAG 有显著优势

## 5. 实践建议

作者提出的渐进式实施策略：

1. **起步阶段:** 混合向量+图谱检索（Hybrid Vector + Graph Retrieval）
2. **优化阶段:** 实现自修复 Cypher 生成（Self-healing Cypher Generation），可将准确率从约 48% 提升至约 63%
3. **进阶阶段:** 根据实际查询模式逐步添加专用组件，而非基于理论需求过度设计

## 6. 核心洞察

- 不存在"银弹"架构，选择应基于实际查询分布
- 混合方案往往是最务实的起点
- 图谱质量（实体链接、关系抽取）是所有图谱RAG方案的基础
- 索引成本和更新频率是生产环境中不可忽视的因素
- 渐进式演进优于一步到位的复杂架构
