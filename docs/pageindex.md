# PageIndex: 无向量、基于推理的下一代 RAG

> 来源: [PageIndex: Next-Generation Vectorless, Reasoning-based RAG](https://pageindex.ai/blog/pageindex-intro)
> 作者: Mingtian Zhang, Yu Tang, PageIndex Team
> 发布日期: 2025-09-19

## 1. 研究动机

LLM 面临上下文窗口限制和长文档性能退化问题。传统基于向量的 RAG 依赖语义相似度，但语义相似并不总意味着真正相关。

### 向量 RAG 的核心限制

| 问题 | 描述 |
|------|------|
| 查询-知识空间错位 | 用户意图（query space）与文档内容（knowledge space）不在同一语义空间 |
| 相似≠相关 | 语义相似度不能保证信息真正回答问题 |
| 硬切分破坏语义 | 固定长度分块（chunking）打断文档上下文和含义 |
| 无法集成对话历史 | 无法利用多轮对话上下文改善检索 |
| 文档内交叉引用处理差 | 无法跟踪文档内部的引用关系 |

## 2. 核心方法：推理式检索

PageIndex 用**动态 LLM 推理**替代静态向量嵌入，通过导航文档结构并推理真正的相关性来实现检索。

### 2.1 五步迭代检索过程

1. **阅读目录:** 理解文档整体结构
2. **选择最相关章节:** 基于推理判断哪个部分最可能包含答案
3. **提取相关信息:** 从选定章节中提取关键内容
4. **评估充分性:** 判断已提取信息是否足以回答问题；不足则循环
5. **生成回答:** 基于收集的信息综合回答

### 2.2 核心创新：JSON 层次索引结构

构建**上下文内索引（In-Context Index）**——驻留在 LLM 推理上下文中的层次结构，而非外部向量数据库。

每个索引节点包含：
- ID
- 名称
- 描述
- 元数据
- 子节点

形成可导航的文档层次树，LLM 通过逐层推理定位信息。

## 3. 向量 RAG vs 推理 RAG 对比

| 维度 | 向量 RAG | 推理 RAG (PageIndex) |
|------|----------|---------------------|
| 上下文推理 | 基于表面语义匹配 | 理解深层含义和上下文 |
| 语义相关性 | 可能返回相似但不相关的结果 | 推理真正的相关性 |
| 连贯段落检索 | 返回碎片化块 | 检索完整的语义单元 |
| 多轮对话感知 | 无法利用对话历史 | 集成对话上下文改善检索 |
| 交叉引用跟踪 | 无法跟踪 | 可以跟踪文档内部引用 |

## 4. 应用场景

- **金融报告分析:** 复杂的跨表格、跨章节信息关联
- **法律文档审查:** 需要理解条款间的引用关系
- **技术手册解读:** 层次化的操作步骤和参数说明
- **多轮文档探索:** 基于对话历史的渐进式深入

## 5. 技术架构特点

| 特点 | 说明 |
|------|------|
| 无向量数据库 | 不需要嵌入模型和向量存储 |
| 结构感知 | 利用文档原生结构（标题、章节、子章节） |
| 迭代式 | 不是一次性检索，而是逐步深入 |
| 推理驱动 | 每一步检索决策都由 LLM 推理指导 |
| 上下文内 | 索引存在于 LLM 上下文中，无外部依赖 |

## 6. 核心洞察

PageIndex 代表了 RAG 的一个**范式转变**：从"检索然后生成"变为"推理式导航"。核心认知是：

1. **检索本身可以是推理任务**——不需要预计算的向量，LLM 可以通过理解文档结构来定位信息
2. **文档结构是被低估的信号**——标题层次、目录结构包含丰富的语义信息，传统 RAG 完全忽视了这些
3. **迭代优于一次性**——像人类翻阅书籍一样逐步缩小范围，比一次性匹配更精准

**局限性考虑:**
- 每次查询需要多轮 LLM 调用，延迟和成本较高
- 依赖文档具有良好的层次结构（对非结构化文本可能不适用）
- 对 LLM 推理能力有较高要求
- 可能不适合需要跨越大量文档的聚合查询

**与向量 RAG 的互补性:** PageIndex 更适合单文档深度理解，而向量 RAG 更适合跨文档的广度检索。实际应用中两者可能互补使用。
