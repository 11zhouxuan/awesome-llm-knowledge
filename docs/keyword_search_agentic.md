# Keyword Search is All You Need: 无向量数据库实现 RAG 级性能

> 论文: [Keyword search is all you need: Achieving RAG-Level Performance without vector databases using agentic tool use](https://arxiv.org/html/2602.23368v1/)
> 作者: Shreyas Subramanian, Adewale Akinfaderin, Yanyan Zhang, Ishan Singh, Mani Khanuja, Sandeep Singh, Maira Ladeira Tanke
> 机构: Amazon Science
> 日期: 2026-02

## 1. 研究动机

向量数据库在 RAG 系统中被视为标配，但本文质疑其必要性：**在智能体框架中，基于工具的关键词搜索是否能达到传统 RAG 系统 90% 以上的性能，同时无需常驻向量数据库？**

核心论点：对于许多应用场景，智能体式关键词搜索可以替代向量 RAG，且更简单、更易维护。

## 2. 核心方法

### 2.1 算法：Agentic Document-Based QA

**核心流程:**

1. **元数据扫描:** 通过 shell 脚本扫描文档元信息
2. **迭代关键词/正则搜索:** 使用 rga、pdfgrep 等工具
3. **逐步上下文扩展:** 通过 Linux shell 命令逐步扩大搜索范围
4. **答案构建:** 基于收集的上下文生成回答

**关键参数:**
- Temperature: 0.001（确保可复现性）
- 最大迭代次数约束（控制搜索深度）

### 2.2 工具集

| 工具 | 用途 |
|------|------|
| rga (ripgrep-all) | 在多种文件格式中进行正则搜索 |
| pdfgrep | PDF 文件内关键词搜索 |
| shell scripts | 元数据扫描和文件导航 |
| Linux 命令 | 上下文提取和扩展 |

### 2.3 智能体循环

```
While (未达到最大迭代 AND 答案不充分):
    1. 基于当前理解生成搜索策略
    2. 执行关键词/正则搜索
    3. 评估搜索结果
    4. 扩展或精炼搜索范围
    5. 积累上下文
Return: 基于累积上下文的答案
```

## 3. 实验结果

### 3.1 性能对比（Agent vs RAG 的达成率）

| 指标 | 达成率 (Agent/RAG) |
|------|-------------------|
| 忠实度 (Faithfulness) | **94.52%** |
| 上下文召回 (Context Recall) | **88.05%** |
| 答案正确性 (Answer Correctness) | **91.48%** |

### 3.2 分数据集结果

| 数据集 | Agent 答案正确性 | RAG 答案正确性 | 达成率 |
|--------|-----------------|---------------|--------|
| BlockchainSolana | - | - | **99.97%** |
| LLM Survey | - | - | **99.51%** |
| FinanceBench | **30.40%** | 24.24% | **>100%（超越）** |

**FinanceBench 案例:** 智能体方法实际上**超越**了传统 RAG，达到 30.40% vs 24.24%。

### 3.3 整体结论

> "基于工具的关键词搜索实现在智能体框架中，可以达到传统 RAG 系统 90% 以上的性能指标，且无需常驻向量数据库。"

## 4. 优势分析

| 维度 | 智能体关键词搜索 | 传统向量 RAG |
|------|-----------------|-------------|
| 基础设施 | 无需向量数据库 | 需要向量DB + 嵌入模型 |
| 维护成本 | 低（文件更新即可） | 高（需要重新索引） |
| 实现复杂度 | 简单（shell工具） | 中等（嵌入+索引+检索） |
| 知识更新 | 即时（文件更新即生效） | 需要重新嵌入和索引 |
| 可解释性 | 高（可追踪搜索路径） | 低（向量相似度不透明） |

## 5. 局限性

| 局限 | 描述 |
|------|------|
| 复杂文档 | 对结构复杂的文档性能下降 |
| 多媒体内容 | 无法处理图片、表格等非文本内容 |
| 上下文细微差别 | 对跨学科内容（如 Paul Graham 散文）理解不足 |
| 延迟 | 多轮迭代搜索比一次向量检索慢 |
| 语义理解 | 纯关键词无法捕获同义词和语义关联 |

## 7. 核心洞察

Amazon Science 的这篇论文提出了与 PwC 的"Is Grep All You Need?"**高度互补**的观点：

1. **向量数据库不是必需品** ——对于频繁更新的知识库，简单的关键词搜索+智能体循环可能是更务实的选择。消除了嵌入计算、向量索引维护和数据库运维的成本。

2. **智能体循环弥补了关键词的局限** ——单次关键词搜索确实弱于向量检索，但通过迭代搜索、上下文扩展和搜索策略调整，智能体可以逼近向量 RAG 的效果。

3. **90% 达成率的务实意义** ——对于许多企业场景，90% 的性能换取极大简化的基础设施，是非常划算的权衡。

4. **FinanceBench 超越** ——在某些数据集上反超 RAG，说明向量检索的"语义噪声"在特定场景下反而是劣势。

**实践启示:**
- 对于频繁更新的文档集（如技术手册、政策文件），智能体关键词搜索可能优于传统 RAG
- 作为 RAG 系统的后备或降级方案非常合适
- 适合 MVP 阶段快速验证，无需投入向量基础设施
- 对于结构化/半结构化文档（如金融报告）表现特别好
