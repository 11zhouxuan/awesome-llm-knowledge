# LLM Wiki: 增量式 Wiki 知识库构建系统

> 项目: [nashsu/llm_wiki](https://github.com/nashsu/llm_wiki)
> Stars: 13.1k | Forks: 1.6k | Releases: 47
> 技术栈: TypeScript (81.1%), Rust (14.2%)
> 协议: GNU GPLv3

## 1. 项目概述

LLM Wiki 是一个跨平台桌面应用，将文档自动转化为有组织的、互相链接的知识库。核心理念来自 Andrej Karpathy 的 LLM Wiki 模式：**不是每次查询时重新推导答案（RAG），而是增量构建和维护一个持久化的 Wiki**。

### 1.1 核心理念对比

| 维度 | 传统 RAG | LLM Wiki 模式 |
|------|----------|--------------|
| 知识存储 | 向量化的文档块 | 结构化的 Wiki 页面 |
| 知识关联 | 隐式（向量相似度） | 显式（Wikilink 互联） |
| 知识更新 | 重新嵌入 | 增量更新页面 |
| 可读性 | 对人不友好 | Markdown，人机可读 |
| 知识发现 | 被动检索 | 主动发现关联和缺口 |

## 2. 系统架构

### 2.1 三层结构

```
┌─────────────────────────────────┐
│       Wiki Layer                │  LLM 生成的页面 + 交叉引用
├─────────────────────────────────┤
│       Schema Layer              │  结构规则和配置
├─────────────────────────────────┤
│       Raw Sources Layer         │  不可变原始文档
└─────────────────────────────────┘
```

- **Raw Sources:** 原始文档（不可变），作为知识的事实来源
- **Wiki:** LLM 生成的结构化页面，带有交叉引用链接
- **Schema:** 控制 Wiki 组织结构的规则和配置

### 2.2 核心操作

| 操作 | 功能 |
|------|------|
| Ingest | 将源文档转化为 Wiki 页面 |
| Query | 基于 Wiki 知识回答问题 |
| Lint | 维护 Wiki 一致性和质量 |

### 2.3 输出格式

- `index.md` — Wiki 索引
- `log.md` — 变更日志
- Wikilink 语法 — 与 Obsidian 兼容

## 3. 关键技术特性

### 3.1 摄入管道（Ingest Pipeline）

| 特性 | 实现 |
|------|------|
| 两步 CoT 处理 | 分析 → 生成 |
| 增量缓存 | SHA256 哈希，未变文件不重处理 |
| 崩溃恢复 | 持久化队列 + 进度可视化 |
| 文件夹导入 | 保留目录结构 |
| 自动监控 | 源文件夹变化自动检测 |

### 3.2 知识管理

**四信号相关性模型:**

| 信号 | 权重 | 描述 |
|------|------|------|
| 直接链接 | ×3.0 | Wikilink 直连 |
| 源文件重叠 | ×4.0 | 共享原始文档来源 |
| Adamic-Adar | ×1.5 | 图论相似度指标 |
| 类型亲和 | ×1.0 | 相同类型的页面 |

**社区检测:**
- Louvain 算法自动聚类
- sigma.js 图可视化，12 色社区分离
- 识别意外关联和知识缺口

### 3.3 检索系统

| 检索方式 | 实现 |
|----------|------|
| 分词搜索 | 支持 CJK 双字符切分 |
| 向量语义搜索（可选） | LanceDB + OpenAI 兼容端点 |
| 上下文窗口 | 可配置 4K-1M tokens |

**上下文预算分配:**
- 60% — Wiki 页面
- 20% — 对话历史
- 15% — 系统提示
- 5% — 索引

### 3.4 多格式支持

PDF（可选 MinerU 解析）、DOCX、PPTX、XLSX、图片

### 3.5 集成能力

- Chrome 扩展 Web Clipper（Readability.js 提取）
- 深度研究（Tavily / SerpApi / SearXNG）
- 本地 HTTP API（端口 19828）
- MCP Server — 供其他 AI 智能体集成
- 异步审查系统

## 7. 核心洞察

LLM Wiki 代表了**从"即时检索"到"持久知识"的范式转换**：

### 7.1 为什么 Wiki 优于 RAG？

1. **知识复用:** RAG 每次查询都重新推导，Wiki 只构建一次。如果 100 人问同一个问题，RAG 做 100 次推导，Wiki 做 1 次构建 + 100 次查找。

2. **显式关联:** Wikilink 使知识关系可见、可审计。RAG 的向量相似度是黑盒，无法解释为什么两个概念相关。

3. **知识发现:** 图可视化和社区检测能主动发现"你不知道自己需要知道的东西"——RAG 只能被动回答已知问题。

4. **人机协同:** Wiki 对人类可读可编辑，团队可以审查和修正 LLM 生成的知识。RAG 的向量索引对人类不透明。

### 7.2 局限性

- 初始构建成本高（需要 LLM 处理所有文档）
- Wiki 页面可能包含 LLM 幻觉（需要审查机制）
- 不适合高度动态的数据源（如实时数据流）
- 对 LLM 质量依赖度高

### 7.3 对行业的意义

13.1k stars 证明了市场对"持久化知识库"的强烈需求。这不只是一个工具——它验证了一种**新的知识管理范式**：LLM 不仅是"回答器"，更是"知识构建者"。Google 的 Open Knowledge Format (OKF) 正在将这一模式标准化，预示着从个人工具到行业标准的演进路径。
